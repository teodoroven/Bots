# Bots (VK + Telegram)

Python-проект с единой абстракцией для разработки ботов под **VK** и **Telegram**.

Проект решает практическую задачу: вместо поддержки двух разных реализаций для каждой платформы он выносит общую бизнес-логику в единый слой и работает через унифицированные сущности сообщений, вложений, событий и клавиатур. Такой подход упрощает масштабирование, повторное использование кода и сопровождение интеграций.

## Зачем этот проект

В реальных задачах ботам часто нужны одинаковые сценарии на разных платформах: принять сообщение, обработать текст и вложения, показать кнопки, отредактировать ответ, сохранить данные, обработать голос и медиа.  
Вместо дублирования логики под каждую платформу я реализовал единый уровень абстракции, который позволяет:

- работать с **VK** и **Telegram** через общую модель;
- переиспользовать бизнес-логику без привязки к конкретному API;
- унифицировать обработку текста, вложений, reply/forward-сценариев и callback-кнопок;
- централизовать вспомогательные функции: логирование, JSON-хранилища, файловые операции.

## Что реализовано

### Основные возможности

- единая модель для работы с ботами двух платформ;
- общие сущности:
  - `Bot`
  - `Event`
  - `Message`
  - `Attachment`
  - `Keyboard`
- поддержка вложений:
  - изображения
  - видео
  - аудио
  - документы
- поддержка reply- и forward-сценариев;
- подготовка и отправка сообщений с клавиатурами;
- редактирование и удаление сообщений;
- обработка callback-сценариев через набор констант;
- работа с голосовыми сообщениями и аудио;
- вспомогательные утилиты для JSON, файлов и логирования;
- базовое покрытие unit-тестами для утилитарного слоя.

### Архитектурная идея

Проект построен вокруг **унифицированного интерфейса сообщений**.  
Это позволяет описывать прикладную логику один раз, а платформенные различия инкапсулировать в реализациях для:

- `Vkbot`
- `Telebot`

На уровне архитектуры это даёт:

- меньше дублирования кода;
- более понятную поддержку новых сценариев;
- удобную точку расширения для новых платформ или типов вложений;
- лучшую тестируемость вспомогательных модулей.

## Ключевые инженерные решения

### 1. Единый слой абстракции над платформами
Вместо двух независимых реализаций проект использует общие модели сообщений и событий, что снижает связность и упрощает сопровождение.

### 2. Работа с медиа и голосом
Поддержана подготовка аудио и файловых вложений, а также база для обработки голосовых сообщений и распознавания речи.

### 3. Выделение инфраструктурного слоя
Логирование, работа с JSON-хранилищами и файловые операции вынесены в отдельный модуль, чтобы не смешивать инфраструктуру и бизнес-логику.

### 4. Проверяемые утилиты
Для вспомогательных функций добавлены unit-тесты: чтение/запись файлов, работа с JSON и путями.

```bash
pip install -r requirements.txt
```
